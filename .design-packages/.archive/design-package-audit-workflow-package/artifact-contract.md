# Artifact Contract

This document defines the bounded output contract for
`/audit-design-package`. It is the package-local source of truth for bundle
layout, report shapes, and mutation receipts.

## Runtime Alignment Rule

- The canonical bundle root for this package is
  `.harmony/output/reports/workflows/YYYY-MM-DD-audit-design-package-<slug>/`.
- The canonical top-level summary path is
  `.harmony/output/reports/YYYY-MM-DD-audit-design-package[-<collision-index>].md`.
- If this document conflicts with the implemented workflow contract at
  `/.harmony/orchestration/runtime/workflows/audit/audit-design-package/workflow.yml`,
  treat the conflict as a readiness blocker and update this package before it is
  declared `implementation-ready`.

## Bundle Root

Each run writes one bounded bundle at:

```text
.harmony/output/reports/workflows/YYYY-MM-DD-audit-design-package-<slug>/
```

The bundle root must contain these required artifacts:

- `bundle.yml` — machine-readable run metadata and outcome
- `plan.md` — resolved execution plan with selected mode and stage set
- `validation.md` — done-gate checklist and final outcome
- `package-delta.md` — aggregate changed/reviewed package summary across
  file-writing stages
- `reports/` — one markdown report per selected stage
- `stage-inputs/` — persisted prompt packets passed to the executor
- `stage-logs/` — executor and runner logs for each selected stage

Optional bundle artifacts may also exist:

- `inventory.md` — bundle-local artifact inventory generated by the runner
- `summary.md` — bundle-local synthesis written by the reporting stage
- `commands.md` — command/executor receipts when the runner emits them
- additional runner support files that do not redefine the required contract

## Directory Contract

The bundle is bounded when every required artifact is present and every optional
artifact lives under the same bundle root.

### `reports/`

The `reports/` directory uses these file names:

- `01-design-package-audit.md`
- `02-design-package-remediation.md` (`short` mode only)
- `03-design-red-team.md` (`rigorous` mode only)
- `04-design-hardening.md` (`rigorous` mode only)
- `05-design-integration.md` (`rigorous` mode only)
- `06-implementation-simulation.md`
- `07-specification-closure.md`
- `08-minimal-implementation-architecture-blueprint.md`
- `09-first-implementation-plan.md`

### `stage-inputs/`

Each selected stage must persist exactly one prompt packet named:

```text
<stage-ordinal>-<stage-id>-<stage-slug>.prompt.md
```

Each prompt packet must contain:

- workflow/stage metadata
- target package path
- prompt source path
- injected prior reports or bundle inputs
- the fully rendered canonical prompt body passed to the executor

### `stage-logs/`

Each selected stage must persist exactly one log named:

```text
<stage-ordinal>-<stage-id>-<stage-slug>.log
```

Each stage log must capture:

- executor selection
- command or invocation mode
- start/end timestamps when available
- exit classification (`success`, `executor-environment-failure`,
  `prompt-packet-failure`, `stage-validation-failure`,
  `package-mutation-failure`, `standard-validator-failure`, `cancelled`, or
  `not-run`)
- enough stderr/stdout context to debug the stage result

## Top-Level Summary

Each run also writes one summary report under `.harmony/output/reports/`.

The summary must include:

- target package path
- selected mode
- bundle root
- final readiness verdict
- highest-severity unresolved blockers
- changed package files
- links or references to the bounded bundle contents

## Report Content Requirements

Every stage report must include these sections:

- `# <Stage Report Title>`
- `## Stage Metadata`
- `## Inputs`
- `## Findings` or `## Decisions`
- `## Next Handoff`

File-writing stage reports must also include:

- `## CHANGE MANIFEST`
- created, updated, removed, and reviewed file lists
- one receipt:
  - `Direct Edit Receipt` when the package was mutated in place, or
  - `Zero-Change Receipt` when no mutation was required

The specification-closure report may use `Zero-Change Receipt` only when stage
`06` identifies zero remaining implementation blockers.

## Machine-Readable Artifact Schemas

The contracts below are normative even when the artifacts are rendered as human
readable markdown.

### `bundle.yml`

`bundle.yml` must record at least:

```yaml
workflow_id: audit-design-package
package_path: /absolute/path/to/package
mode: rigorous
executor: codex
slug: stable-output-slug
bundle_root: /absolute/path/to/bundle
summary_report: /absolute/path/to/summary-report.md
selected_stages:
  - id: "01"
    report_path: reports/01-design-package-audit.md
    class: evaluative
report_paths:
  "01": reports/01-design-package-audit.md
changed_files:
  - stage: "04"
    path: artifact-contract.md
    action: update
final_verdict: in-review
run_status: completed
```

Required invariants:

- `workflow_id` must be `audit-design-package`
- `package_path`, `bundle_root`, and `summary_report` must be absolute paths
- `report_paths` and `selected_stages` must describe the same selected stage set
- `changed_files` must match the union of file-writing stage manifests
- `final_verdict` must be explicit

### `validation.md`

`validation.md` must evaluate at least:

```yaml
selected_mode: rigorous
expected_stage_reports:
  - reports/01-design-package-audit.md
present_stage_reports:
  - reports/01-design-package-audit.md
mutation_receipts:
  "04": direct-edit
  "05": zero-change
top_level_summary_present: true
package_validator:
  required: true
  result: pass
final_verdict: in-review
bundle_valid: true
```

Required invariants:

- stage coverage matches the selected mode
- required report files exist
- required file-writing stages include a change manifest or zero-change receipt
- top-level summary exists
- final verdict is explicit

### `package-delta.md`

`package-delta.md` must aggregate all reviewed or changed package files for the
run and include:

- the target package path
- stage-by-stage mutation summary
- all created, updated, removed, and reviewed files
- unresolved follow-up items, if any

### `CHANGE MANIFEST`

Every file-writing stage report must include a `CHANGE MANIFEST` with these
fields:

```yaml
stage: "04"
mutation_status: changed
created_files:
  - normative/execution/run-lifecycle.md
updated_files:
  - artifact-contract.md
removed_files: []
reviewed_files:
  - README.md
direct_edit: true
patch_output_required: false
```

### `Zero-Change Receipt`

A zero-change receipt is valid only when it contains:

```yaml
stage: "07"
mutation_status: zero-change
rationale: stage-06 reported no remaining blockers
files_reviewed:
  - artifact-contract.md
reason_no_mutation_required: package already satisfies the stage contract
```
