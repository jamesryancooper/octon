# Migration Plan

## Phase 1: Proposal Packet

Create and validate this architecture proposal under:

```text
.octon/inputs/exploratory/proposals/architecture/mission-plan-compiler-layer/
```

This phase creates no runtime authority and no material planning control state.

## Phase 2: Framework Doctrine And Schemas

Promote accepted doctrine and schemas under:

```text
.octon/framework/engine/runtime/spec/
```

Update structural and constitutional registries so the new path families,
control roots, evidence roots, validators, and generated projection posture are
declared.

## Phase 3: Workflow Integration

Add the mission workflow:

```text
.octon/framework/orchestration/runtime/workflows/missions/derive-mission-plan/
```

The workflow should bind to mission authority, run critic checks, selectively
decompose branches, compile ready leaves, and update from retained evidence.

## Phase 4: Instance Policy

Add optional instance enablement:

```text
.octon/instance/governance/policies/hierarchical-planning.yml
```

The policy should set default budgets, risk thresholds, approval thresholds,
and whether planning is enabled for a mission class.

## Phase 5: State Roots

Only after framework and validator promotion, allow mission-bound control and
evidence roots:

```text
.octon/state/control/execution/missions/<mission-id>/plans/<plan-id>/
.octon/state/evidence/control/execution/planning/<plan-id>/
```

## Phase 6: Generated Operator Views

Generated planning views may be added only after source traceability,
freshness, and non-authority metadata are enforced.

## Phase 7: Closeout

Promotion closeout must update this packet's conformance and drift/churn
receipts, remove active proposal-path dependencies from durable targets, and
archive the packet with promotion evidence.
