# Reusable Patterns

- proposal: `octon-proposal-packet-lifecycle-automation`

## Purpose

This proposal adopts a small set of reusable implementation patterns for the
`octon-proposal-packet-lifecycle` extension pack. These patterns prevent the
automation from becoming a loose collection of prompts and make the lifecycle
testable, routable, and safe to publish.

After implementation, the durable home for these patterns is the extension
pack, preferably:

```text
.octon/inputs/additive/extensions/octon-proposal-packet-lifecycle/context/patterns.md
```

If the material grows, the pack may split them into:

```text
.octon/inputs/additive/extensions/octon-proposal-packet-lifecycle/context/patterns/
```

This proposal-local file describes and justifies the intended design. It is not
the long-term authority after promotion.

## Pattern 1 - Lifecycle State Machine

### Shape

Model packet automation as explicit states and transitions:

```text
source-context
  -> packet-created
  -> packet-validated
  -> implementation-prompt-generated
  -> implemented
  -> verification-prompt-generated
  -> verified
  -> corrections-needed
  -> corrected
  -> clean
  -> closeout-prompt-generated
  -> closeout-ready
  -> archived
```

The state model must also allow fail-closed terminal or pause states:

- `blocked`
- `needs-packet-revision`
- `superseded`
- `explicitly-deferred`

### Justification

The manual lifecycle is currently implicit in prompt sequencing. A state
machine gives the composite route a deterministic way to decide which bundle is
valid next and prevents accidental jumps from packet creation directly to PR
merge.

## Pattern 2 - Route Dispatcher

### Shape

Expose one composite entry point that routes to leaf bundles using:

- source kind,
- proposal kind,
- lifecycle action,
- packet path or proposal id,
- current lifecycle state,
- user constraints,
- available existing Octon route coverage.

### Justification

The lifecycle has many entry points: audit, evaluation, drift report, existing
packet, verification finding, or closeout request. A dispatcher keeps the
operator interface stable while allowing leaf bundles to remain narrow and
testable.

## Pattern 3 - Packet Support Artifact Placement

### Shape

Preserve source lineage and generated operational prompts in stable packet
locations:

```text
resources/
  source-context.md
  source-evaluation.md
  source-audit.md

support/
  proposal-packet-creation-prompt.md
  executable-implementation-prompt.md
  follow-up-verification-prompt.md
  custom-closeout-prompt.md
  correction-prompts/
```

### Justification

The manual prompts repeatedly require full audit/evaluation context and custom
prompts to stay with the packet. This pattern makes that retention predictable
without letting support artifacts become authority.

## Pattern 4 - Finding To Correction

### Shape

Verification findings must have stable identities and enough context to create
targeted correction prompts:

- stable finding id,
- severity,
- affected paths,
- evidence,
- expected behavior,
- correction scope,
- acceptance criteria,
- deferral eligibility.

Each unresolved finding should generate one correction prompt unless a finding
group is explicitly justified.

### Justification

Manual correction is currently ad hoc. Stable finding identities make repeated
verification meaningful and make correction prompts small enough to execute
without broadening scope.

## Pattern 5 - Convergence Loop

### Shape

Run verification and correction until one of these outcomes is reached:

- `clean`
- `blocked`
- `needs-packet-revision`
- `superseded`
- `explicitly-deferred`

The loop must record each verification pass and correction prompt.

### Justification

The manual process says to verify complete implementation and correct remaining
findings. A convergence loop makes this repeatable and prevents premature
closeout after the first partial fix.

## Pattern 6 - Closeout Gate

### Shape

Closeout is a gated lifecycle route, not a final chat response. It must verify:

- proposal promoted or archived correctly,
- proposal registry regenerated by the canonical generator when safe,
- housekeeping completed before staging,
- incidental build and scratch artifacts excluded,
- intended files staged,
- commit and PR created,
- required checks green,
- review conversations resolved,
- branch merged only after gates pass,
- local and origin branches cleaned up and synced.

### Justification

The manual closeout prompt spans proposal lifecycle, Git, GitHub, CI, review,
merge, branch cleanup, and sync. Treating it as a gate protects against
half-closed proposal work and accidental merge with failing checks or unresolved
reviews.

## Pattern 7 - Evidence Receipt

### Shape

Each lifecycle route emits or updates retained evidence:

- route input summary,
- selected route and rationale,
- source or packet path,
- generated artifact paths,
- commands run,
- validation results,
- findings and correction status,
- final route verdict.

Use existing evidence and checkpoint roots instead of inventing a new evidence
plane.

### Justification

Proposal lifecycle automation is only useful if later agents can determine
what happened and why. Evidence receipts make the automation observable without
making proposal packets or generated prompts authoritative.

## Pattern 8 - Composition First

### Shape

Prefer existing Octon surfaces before custom behavior:

- proposal create workflows,
- proposal validators,
- proposal registry generator,
- concept-integration source-to-packet routes,
- concept-integration packet-to-implementation route,
- impact-map-and-validation-selector,
- drift-triage,
- retirement-and-hygiene packetizer,
- extension publication scripts,
- capability and host projection publication scripts.

### Justification

Octon already has proposal and extension infrastructure. The new pack should
compose that infrastructure into a lifecycle, not duplicate it.

## Pattern 9 - Authority Firewall

### Shape

Every route and generated prompt must preserve these non-authority boundaries:

- prompts do not become authority,
- generated support artifacts do not become authority,
- proposal packets remain temporary and non-canonical,
- generated proposal registry is discovery-only,
- GitHub comments, PR labels, CI dashboards, chat, model memory, MCP/tool
  availability, browser state, and external dashboards do not become Octon
  authority, control truth, runtime policy, or permission.

### Justification

The manual prompts repeatedly emphasize promotion safety and no rival control
plane. The authority firewall turns that into a pack-wide rule.

## Pattern 10 - Scenario Fixtures

### Shape

Every manual prompt class must have at least one validation fixture:

- audit-aligned packet creation,
- concise packet creation,
- architecture evaluation packet creation,
- highest-leverage next-step packet creation,
- packet explanation,
- implementation prompt generation,
- evaluation prompt refresh,
- follow-up verification prompt generation,
- closeout prompt generation and gate behavior.

### Justification

The automation must be testable against the real manual workflow it replaces.
Scenario fixtures make regressions visible when prompt bundles, route contracts,
or shared patterns change.

## Pattern 11 - Proposal Program

### Shape

Coordinate a set of related child proposal packets from a parent program
packet. The parent is itself a normal proposal packet and owns sequence,
dependencies, aggregate implementation prompts, aggregate verification,
aggregate closeout, cross-packet risks, and program evidence.

Child packets remain normal first-class proposals at canonical paths:

```text
.octon/inputs/exploratory/proposals/<kind>/<child-proposal-id>/
```

Child packets must not be nested under the parent packet directory.

The full pattern is specified in `architecture/proposal-program-pattern.md`.
After implementation, its durable home should be inside the extension pack
context tree, preferably as `context/patterns/proposal-program.md`, and it must
be exposed through program routes for creation, implementation prompt
generation, verification prompt generation, correction prompt generation,
verification/correction convergence, closeout prompt generation, and closeout
execution.

### Justification

Some Octon initiatives are too large or too heterogeneous for a single
proposal packet but still need one governed implementation and closeout
program. This pattern supports parent-driven sequencing without weakening the
existing rule that every proposal packet has its own canonical path,
manifest authority, subtype contract, validators, and promotion targets.

## Implementation Rule

The extension pack must expose these patterns to prompt bundles through shared
context or references. Route-specific prompt bundles should import or cite the
relevant patterns instead of restating them independently.
