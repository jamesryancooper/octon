# Acceptance Criteria

The implementation is accepted only when:

- New contracts exist under durable framework surfaces.
- Repo-specific self-evolution policies exist under instance governance.
- Evolution Candidates, simulations, amendment requests, promotions, recertifications, and ledger have canonical control/evidence placement.
- Proposal compiler output satisfies live proposal standards and remains non-authoritative.
- Promotion can only write accepted proposal outputs to declared durable targets.
- Promotion emits retained receipts.
- Recertification is required before closure.
- Negative-control tests prove evidence, generated summaries, lab success, simulation success, and proposal packets cannot self-authorize authority changes.
- Runtime/CLI surfaces either implement the v5 flow or fail closed with explicit reason codes.
- Documentation states actual implemented behavior and deferred scope.
