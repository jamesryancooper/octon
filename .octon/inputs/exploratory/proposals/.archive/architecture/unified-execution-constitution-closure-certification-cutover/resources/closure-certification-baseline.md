# Closure Certification Baseline

This packet treats the next round as a **closure certification round**, not
another architecture round.

## Why the posture changes now

Octon is no longer missing the target-state nouns. It already has:

- a supreme constitutional kernel
- explicit normative and epistemic precedence
- an active constitutional contract registry
- published support targets
- non-authoritative host/model adapters
- first-class `lab/**` and `observability/**` domains
- canonical per-run control roots
- retained RunCard/HarnessCard disclosure roots
- an existing release HarnessCard that already claims an atomic execution
  constitution cutover in bounded form

That changes the problem. The question is no longer “what architecture should we
add?” It is “what precise claim can the repository prove, and what remains only
declared?”

## The critical normalization

The claim must be **bounded correctly**.

The support-target matrix already makes clear that the clean supported envelope
is centered on:

- `MT-B`
- `WT-2`
- `LT-REF`
- `LOC-EN`

GitHub is currently reduced. CI is currently projection-only or stage-only.
Other combinations remain reduced, experimental, or denied.

Therefore the honest closure claim is:

> **Octon is a fully realized unified execution constitution within its declared
> supported envelope.**

It is **not**:

> “every host, workload, locale, and adapter surface is fully realized.”

## Residual mismatch that must be closed

The most important remaining mismatch is host-native governance. The GitHub host
adapter contract says GitHub is non-authoritative and projects labels/comments/
checks without minting authority. Closure still requires the practical workflow
paths to behave that way.

Specifically:

- workflows may project status
- workflows may materialize canonical decision or approval artifacts by calling
  canonical logic
- workflows may not remain the final source of authority

## Required closure moves

1. Freeze the exact claim in one machine-readable closure manifest.
2. Convert residual host-native governance into canonical authority artifacts.
3. Make the full consequential run bundle a hard release gate.
4. Turn support targets into executable positive and negative tests.
5. Make proof planes, lab evidence, and disclosure parity release-blocking.
6. Prove historical shims are no longer path-critical.
7. Make build-to-delete real enough to survive audit.

## Definition of done

Do not exit the round until all of the following are true:

1. Octon’s release claim is explicitly bounded to the already declared supported
   envelope.
2. No GitHub workflow remains a hidden authority source for the certified
   envelope.
3. A supported-envelope consequential run automatically emits the full required
   constitutional bundle.
4. Reduced and unsupported tuples demonstrably stage or deny.
5. Historical shims are either deleted or statically proven non-authoritative.
6. RunCard and HarnessCard claims are release-blocked unless every referenced
   proof artifact resolves.
7. At least one live deletion or demotion receipt proves build-to-delete is not
   just aspirational.

## Certification outcome model

This packet is successful only if the next review becomes binary:

- **pass**: the bounded claim is certified by the proof set
- **fail**: one or more claim-blocking conditions remain unresolved

There is no “almost there” architecture verdict after this packet.
