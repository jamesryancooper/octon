## 1. Executive summary

* I audited the current default branch `main` of `jamesryancooper/octon` and treated the live model as the March 30, 2026 atomic cutover plus the constitutional kernel under `/.octon/framework/constitution/**` and the active workspace charter pair. ([GitHub][1])
* The core governing model is real, not just aspirational prose: the repo currently contains bound run contracts, authority-decision artifacts, retained RunCards, and a release HarnessCard tied to support-target declarations and proof bundles. ([GitHub][2])
* On first principles, Octon is mostly coherent on the things that matter most: constitutional kernel supremacy, authored-vs-derived authority separation, workspace-vs-mission-vs-run distinction, fail-closed routing, role separation, and support-target-bounded consequential execution. ([GitHub][3])
* The ingress/adapter model is aligned: `/.octon/AGENTS.md`, `/AGENTS.md`, and `/CLAUDE.md` are thin adapters pointing back to `/.octon/instance/ingress/AGENTS.md`, and they explicitly deny authority minting at the adapter layer. ([GitHub][4])
* The protected principles surface is also mostly aligned: it explicitly declares itself subordinate to the constitutional kernel, and its substantive content translates the kernel into enforceable engineering habits rather than trying to outrank it. ([GitHub][5])
* There is one serious kernel-level drift incident: `/.octon/framework/constitution/contracts/disclosure/family.yml` still encodes a transitional disclosure model and a lab-local HarnessCard evidence root, which conflicts with the live atomic-cutover model, the disclosure README, the authored HarnessCard source, the retained release disclosure root, and the explicit “historical mirror only” status of lab-local harness cards. ([GitHub][6])
* There is another material drift in bootstrap orientation: `/.octon/instance/bootstrap/START.md` lists `inputs/additive/extensions/**` as instance-native repo authority, which conflicts with the kernel rule that only `framework/**` and `instance/**` are authored authority and that raw `inputs/**` remain non-authoritative. ([GitHub][7])
* Support-target governance is published and wired into authority decisions, but the retained proof I found is narrower than the full published support matrix. The current release HarnessCard proves a repo-local consequential tuple (`repo-shell` + `repo-local-governed` + `LOC-EN`), not the full set of declared host adapters, locale envelopes, or broader portability claims. ([GitHub][8])
* Overall verdict: **substantial alignment with non-trivial drift to clean up**.

## 2. Live Unified Execution Constitution model summary

* `/.octon/framework/constitution/**` is the supreme repo-local control regime beneath non-waivable external obligations, break-glass directives, and live revocations. ([GitHub][3])
* Authored authority is restricted to `framework/**` and `instance/**`; `generated/**` is derived-only; raw `inputs/**` are non-authoritative; `state/**` is operational truth/evidence/control, not authored policy. ([GitHub][3])
* The canonical repo-wide objective surface is the workspace charter pair: `/.octon/instance/charter/workspace.md` plus `workspace.yml`. ([GitHub][9])
* Mission authority remains the continuity container for recurring, overlapping, and long-horizon autonomy; it is not the atomic consequential execution unit. ([GitHub][10])
* Consequential execution is run-contract-first: each consequential run binds through `/.octon/state/control/execution/runs/<run-id>/run-contract.yml`, with stage attempts subordinate under the bound run root. ([GitHub][11])
* Fail-closed routing uses `DENY`, `STAGE_ONLY`, or `ESCALATE` when ownership, evidence, support-targets, run contracts, or policy coherence are missing, or when chat/UI/generated/inputs try to become authority. ([GitHub][12])
* Humans own policy authorship, approvals, exceptions, revocations, support widening, and public commitments; the harness owns routing, validation, evidence, gating, and disclosure; the model owns only bounded planning/execution inside granted scope. ([GitHub][13])
* Normative precedence places the kernel above repo governance declarations, run artifacts, workspace/mission/run layers, and all informative context. ([GitHub][9])
* Epistemic precedence puts live runtime evidence and deterministic validator outputs above current repo state, docs, chat history, and model priors. Runtime-vs-doc mismatches become drift incidents. ([GitHub][14])
* Support-target declarations bound consequential execution across model/workload/context/locale and across host/model adapters; undeclared widening is forbidden and reduced/experimental combinations default to stage-only or deny. ([GitHub][3])
* Disclosure artifacts are subordinate summaries, not authority: RunCards summarize one consequential run; HarnessCards summarize one support/benchmark/release claim; both must be backed by durable authority and retained evidence. ([GitHub][3])
* The live March 30 cutover declares one atomic clean-break model: run-contract-first, runtime-consumed authority, canonical authored + retained disclosure roots, and support-target-enforced operation without transitional live wording. ([GitHub][15])

## 6. Drift and contradiction ledger

1. **Disclosure family drift inside the active constitutional family**

   * **Source paths:** `/.octon/framework/constitution/contracts/disclosure/family.yml` vs `/.octon/framework/constitution/charter.yml`, `/.octon/instance/charter/workspace.{md,yml}`, `/.octon/instance/cognition/context/shared/migrations/2026-03-30-unified-execution-constitution-atomic-cutover/plan.md`, `/.octon/framework/constitution/contracts/disclosure/README.md`, `/.octon/instance/governance/disclosure/README.md`, `/.octon/state/evidence/lab/harness-cards/README.md`. ([GitHub][6])
   * **Conflict:** the active family file still says `change_profile: transitional` and places HarnessCard evidence under `state/evidence/lab/harness-cards`, while the live cutover and disclosure surfaces say the supported model is atomic and canonical disclosure lives under authored governance disclosure plus retained release/run disclosure roots.
   * **Type:** normative drift + stale transition residue.
   * **Why it matters:** this is a kernel-family contradiction, not just an informative doc typo; it can misroute validators, writers, or reviewer expectations toward historical mirror paths.

2. **Bootstrap authority leak into `inputs/**`**

   * **Source paths:** `/.octon/instance/bootstrap/START.md` vs `/.octon/framework/constitution/CHARTER.md`, `/.octon/framework/constitution/charter.yml`, `/.octon/README.md`, `/.octon/framework/cognition/_meta/architecture/specification.md`. ([GitHub][7])
   * **Conflict:** `START.md` lists `inputs/additive/extensions/**` among “instance-native repo authority,” but the kernel and architecture spec say raw `inputs/**` are non-authoritative and trust-agnostic.
   * **Type:** normative drift.
   * **Why it matters:** it blurs the repo’s central authority boundary and could encourage unsupported pack material to be treated as live governance/runtime input.

3. **Family-level profile-selection receipts are not consistently keyed to the final live cutover**

   * **Source paths:** `/.octon/framework/constitution/contracts/{objective,authority,runtime,assurance,retention}/family.yml` vs `/.octon/framework/constitution/charter.yml` and the workspace charter pair. ([GitHub][23])
   * **Conflict:** the active family files point to earlier phase receipts from March 28–29, while the live model receipt named by the charter manifest and workspace pair is the March 30 atomic cutover.
   * **Type:** stale transition residue / ambiguity.
   * **Why it matters:** this may be acceptable as historical activation lineage, but it is not labeled that way; tooling and reviewers could misread phase receipts as the live-model anchor.

4. **Published support matrix outruns current retained disclosure proof**

   * **Source paths:** `/.octon/instance/governance/support-targets.yml` vs current release disclosure root and current release HarnessCard. ([GitHub][8])
   * **Conflict:** the support-target matrix publishes studio/github/ci host adapters, `LOC-MX`, and experimental model envelopes, but the current retained release disclosure packet proves only the repo-local tuple used in the release HarnessCard.
   * **Type:** missing proof.
   * **Why it matters:** a published compatibility matrix without matching disclosure packets invites over-reading of what is actually supported today.

5. **Portability prose outruns repo-local proof**

   * **Source paths:** `CHARTER.md`, workspace charter, `START.md`, principles charter, `.octon/README.md`, root `README.md` vs current retained release disclosure. ([GitHub][3])
   * **Conflict:** purpose/success prose repeatedly claims portability and self-containment across repositories and agent environments, but the retained proof packet I reviewed is still repo-local and adapter-specific.
   * **Type:** missing proof.
   * **Why it matters:** this affects the repo’s stated purpose and operator expectations, especially around support posture and transferability.
