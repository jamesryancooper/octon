# Target-State Specification — Honest Bounded UEC for the Admitted Live Support Universe

## 1. Target-state statement

The target state is:

> **Octon materially substantiates a bounded Unified Execution Constitution for the admitted live support universe, and the active complete claim is derived from validator-backed run evidence, proof-plane coverage, and claim-calibrated release disclosure.**

This target is narrower than “global completion” and stronger than “architecturally close.” It requires **bounded complete closure**.

## 2. Non-negotiable invariants

### 2.1 Constitutional invariants

- `/.octon/framework/constitution/**` remains the singular constitutional kernel.
- Normative and epistemic precedence remain explicit and active.
- No new co-equal charter / constitution surfaces may emerge outside the kernel.

### 2.2 Objective invariants

- `/.octon/instance/charter/**` remains the canonical workspace-charter root.
- `/.octon/instance/orchestration/missions/**` remains the mission continuity container, not the atomic run unit.
- All consequential work binds a per-run objective contract under `/.octon/state/control/execution/runs/**`.
- Stage attempts, retries, checkpoints, rollback posture, and runtime state remain bound to run roots, not mission authority.

### 2.3 Authority invariants

- `approvals/**`, `exceptions/**`, and `revocations/**` remain the sole canonical authority families.
- Host-native labels/comments/checks/env flags may project, request, or summarize — they may never mint authority.
- Claim-bearing live authority artifacts may not contain exercise-only lineage markers.

### 2.4 Runtime invariants

- Every claim-bearing run has a canonical run contract, run manifest, runtime state, authority linkage, checkpoints, replay pointers, trace pointers, and disclosure bundle.
- Resumability must be artifact-driven, not chat-continuity-driven.
- Recovery posture must be explicit and evidenced.

### 2.5 Evidence invariants

- Claim-bearing run evidence must be classed, non-empty, and retention-governed.
- Instruction-layer manifests must describe the actual authoritative layer stack in force for the run.
- Release-level disclosure may only cite validated run bundles.

### 2.6 Disclosure invariants

- RunCards and HarnessCards are canonical disclosure artifacts, not ornamental docs.
- Generated/effective projections remain derived-only and may not be mistaken for authority.
- Historical bundles remain retained evidence only.

## 3. Admitted live support universe — closure-grade interpretation

The repo’s current admitted live universe includes model classes, workload classes, language/context classes, locales, host adapters, model adapters, and capability packs. The complete bounded claim is only honest when every **admitted tuple** needed for the claim is simultaneously:

- formally admitted
- dossier-backed
- adapter-bound
- capability-pack-bounded
- validator-covered
- evidence-backed
- disclosed in the active HarnessCard

### 3.1 Tuple rule

For each admitted tuple:

| Tuple class | Minimum runtime proof | Minimum governance proof | Minimum disclosure |
|---|---|---|---|
| Observe-and-read | read-only exemplar or evaluator-backed scenario | support admission + dossier + host non-authority proof | HarnessCard coverage row |
| Repo-consequential | clean claim-bearing run bundle + functional proof + recovery drill | approval integrity + support dossier + disclosure calibration | HarnessCard row + exemplar RunCard |
| Boundary-sensitive | clean claim-bearing run bundle + functional + behavioral + recovery proof + replay integrity | approval / exception / revocation integrity + host non-authority negative tests + dossier completeness | HarnessCard row + exemplar RunCard + release closure refs |

### 3.2 Unsupported behavior rule

If any tuple is admitted but under-proved at closure time, Octon must either:

1. keep the full universe but downgrade the claim status to recertification-open, or
2. narrow the public complete claim to the fully substantiated subset.

Silent support inflation is prohibited.

## 4. Proof-plane sufficiency standard

A bounded complete claim requires six proof planes. The proof-plane contract is satisfied only when each plane meets **both** of the following:

- **coverage:** all relevant admitted tuples and runtime surfaces are represented
- **disclosure:** the active release bundle makes the plane visible and attributable

### 4.1 Structural proof

Must show:

- contract / schema conformance
- registry consistency
- family completeness
- cross-artifact consistency
- adapter conformance surfaces bound to declared support tuples

### 4.2 Functional proof

Must show:

- intended user-visible behavior works end-to-end on claim-bearing surfaces
- acceptance criteria are executed, not merely narrated
- boundary-sensitive and consequential runs pass their required end-state conditions

### 4.3 Behavioral proof

Must show:

- scenario coverage
- hidden checks
- shadow or replay exercises where applicable
- adversarial / fault exploration for higher-risk classes

### 4.4 Maintainability proof

Must show:

- architecture rules remain enforced
- stale-doc and state-drift detection are live
- structural simplification and anti-slop mechanisms operate continuously
- obsolete scaffolding is reviewed for deletion or retention

### 4.5 Governance proof

Must show:

- authority families are canonical in practice
- workflows do not mint authority
- support-target boundedness is real, not nominal
- disclosure and closure state do not outrun ledgers

### 4.6 Recovery proof

Must show:

- checkpoint / resume works from artifacts
- rollback or compensation posture is explicit and tested
- replay integrity is intact
- contamination / reset handling is defined for claim-bearing classes

## 5. Agency target state

The closure-grade agency posture is:

- a default single accountable orchestrator
- optional overlays allowed only when registered as non-authoritative
- no persona surface may alter governance, support routing, authority, or claim state
- sub-agents may exist when runtime-governed and purpose-bounded; roleplay-heavy authority is forbidden

## 6. Build-to-delete target state

The target state does not require every shim to be deleted immediately. It requires every shim, projection, and historical mirror to be:

- clearly classified,
- non-authoritative in operation,
- covered by retention rationale, and
- actively reviewed for deletion.

## 7. Final closure rule

A complete bounded claim may only be reissued when:

- all closure blockers are resolved,
- all validators pass twice consecutively,
- all claim-bearing runs are bundle-complete,
- all release-level disclosure is regenerated from canonical run bundles, and
- the active release bundle is fresher than all superseded complete-claim bundles.
