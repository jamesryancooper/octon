# Decision Record Plan

## Recommended ADR

If this proposal is accepted, create a durable ADR under:

`/.octon/instance/cognition/decisions/`

Suggested title:

`ADR: Authorized Effect Tokens as Mandatory Material Side-Effect Boundary`

## ADR should record

- why token enforcement follows the Run Journal;
- why token enforcement is a runtime substrate hardening step, not a new Control Plane;
- why material APIs must consume verified tokens;
- why generated/read-model outputs cannot mint or validate tokens;
- why stage-only surfaces remain non-live;
- why negative bypass tests are required for support-target proof.

## ADR is not required before proposal draft use

The packet can be reviewed as a proposal before ADR creation. The ADR should be created when the architecture motion is accepted for promotion.
