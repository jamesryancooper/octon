# Packet Basis Refinements

This resource captures the key refinements that the design packet carries forward beyond the weaker or more provisional elements of the audit.

## 1. Completion architecture, not greenfield redesign

Octon does **not** need a new architecture. It already has the right constitutional superstructure: constitutional kernel, run-oriented control/evidence roots, adapter boundaries, top-level lab, top-level observability, support-target matrix, RunCard/HarnessCard disclosure, and orchestrator-centered agency simplification. The correct move is to **preserve and harden**, not replace.

## 2. Claim truth must invert from closure-local to evidence-first

The repository now contains sophisticated closure and claim artifacts (`claim-truth-conditions.yml`, `closure-summary.yml`, `gate-status.yml`, `phase-status.yml`, and release HarnessCards with completion-shaped claims). The refined design position is that these artifacts must become **generated summaries** of independent evidence rather than first-order truth sources.

**Implication:** closure outputs cannot certify completion on their own. Completion must derive from support-target admissions, proof bundles, retained run evidence, adapter conformance, replay evidence, and retirement status.

## 3. Some audit gaps are narrower than first described

The packet basis carries forward stronger reasoning that the following surfaces already exist in substantive form and therefore need **normalization and binding**, not invention from scratch:

- a live exception-lease set with concrete leases;
- a live stage-attempt artifact for a consequential run;
- workflow-backed simplification / build-to-delete validation hooks.

This does **not** mean the target-state is achieved. It means the remediation posture changes from “create” to “normalize, integrate, and generalize.”

## 4. Present completion may not be claimed

The audit remains authoritative on the central judgment: Octon cannot yet honestly claim that it is already a fully unified execution constitution. The packet basis therefore rejects any prompt language that presumes present target-state attainment, zero unresolved gaps, or immediate closure certification.

## 5. Big-bang migration is architecturally wrong here

The refined packet explicitly rejects the original “single, atomic, big-bang, clean-break migration” framing.

Why:

- the repo already contains substantive live assets that must be preserved;
- some consequential surfaces remain `stage_only` and require bounded transition handling;
- mission/run repair, runtime eventification, authority normalization, and support-target promotion all require staged rollout with explicit truth boundaries.

**Replacement principle:** phased migration with explicit transition modes, compatibility shims where necessary, and strict exclusion of transitional tuples from live completion claims.

## 6. Build-to-delete is a completion property, not post-claim cleanup

The packet basis treats retirement, ablation, and deletion as **constitutional truth conditions**, not optional later work. No compensating mechanism, shim, stage-only exclusion, or legacy persona surface may remain ownerless, uncadenced, or triggerless if Octon is to claim completion.

## 7. Runtime truth needs event-ledger discipline

The repo’s projections (run manifest, runtime state, checkpoints, disclosure) are valuable, but the refined end-state requires an append-oriented event truth model so that runtime state is reconstructible and replayable, rather than merely inspectable via scattered projections.

## 8. Mission / run / stage / attempt must be split cleanly

The packet basis tightens the objective model:

- workspace charter: enduring envelope;
- mission charter: medium-lived governed objective;
- run contract: bounded execution agreement;
- stage contract: ordered slice of a run;
- execution attempt: actual try against a stage.

This cleanly resolves the previous overload of stage and attempt semantics.

## 9. Kernel agency should simplify around real boundary value

The refined packet preserves the orchestrator-centered kernel and keeps verifier-style independence where needed, while demoting or deleting persona-heavy legacy surfaces from the kernel path unless they are still demonstrably load-bearing.

## 10. Support-target honesty is a strength

The support-target matrix and dossier structure should remain narrow and explicit. `stage_only`, `experimental`, or otherwise restricted tuples may exist, but must be clearly excluded from the live completion claim envelope unless and until promotion criteria are actually met.
