# Risk Register

| Risk | Why it matters | Likelihood | Impact | Mitigation |
|---|---|---|---|---|
| Doctor workflow becomes a docs-only checklist | would create pseudo-coverage | medium | high | require real workflow contract, receipt emission, and assurance suite |
| Repo-shell classifier bypasses canonical authorization | would create shadow authority | low | very high | require classifier outcomes to flow through existing authorization/evidence paths only |
| Branch freshness gate lands on only one workflow and gets falsely treated as complete | would leave broad-verification coverage inconsistent | medium | high | packet closure requires coverage across all listed repo-consequential workflows |
| Failure taxonomy expands without workflow emission requirements | would leave operator summaries ad hoc | medium | medium | couple taxonomy extension with reporting policy and workflow output requirements |
| Repo-shell scenario pack exists without runnable workflow | would create scenario-only pseudo-coverage | medium | high | require `/run-repo-shell-supported-scenario` as part of closure |
| Active UEC packet and this sibling packet drift apart | could create proposal lineage confusion | low | medium | cross-reference the active packet in README and baseline audit; keep this packet scoped and sibling-only |
| Support-universe widening slips in through scenario or workflow language | would violate live claim posture | low | very high | explicit no-widening rule in packet manifests and acceptance criteria |
