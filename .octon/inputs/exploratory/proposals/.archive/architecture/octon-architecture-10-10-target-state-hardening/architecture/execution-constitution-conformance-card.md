# Execution Constitution Conformance Card

## Claim

After implementation, Octon's consequential execution conforms to the promoted
constitutional execution model: every material side effect is explicitly scoped,
authorized, fail-closed, observable, reviewable, recoverable, and evidenced.

## Required conformance facts

| Fact | Required evidence |
| --- | --- |
| Material side-effect inventory complete | Inventory validator report. |
| Every material path is authorization-bound | Authorization coverage report and tests. |
| Run contract is atomic execution unit | Run lifecycle fixtures and transition validator. |
| Mission is continuity container | Mission validator and run binding evidence. |
| Support tuple is admitted and live | Live partition admission, dossier, proof bundle, support card. |
| Pack admission does not widen support | Support-pack-admission validator. |
| Generated/effective output is fresh | Publication receipt and freshness validator. |
| Denials and stage-only routes emit reason codes | Negative-control evidence. |
| Operator-visible state is updated | Runtime state and generated read-model traceability. |
| Closure is evidence-complete | RunCard, replay bundle, evidence completeness receipt. |

## Non-conformance conditions

Any of the following blocks conformance:

- missing GrantBundle before material action;
- missing run contract;
- raw input dependency;
- generated artifact as authority;
- host projection as authority;
- stale generated/effective output;
- unadmitted support tuple;
- unadmitted pack in live route;
- hidden human intervention;
- missing support proof bundle;
- missing compatibility shim owner or retirement trigger.
