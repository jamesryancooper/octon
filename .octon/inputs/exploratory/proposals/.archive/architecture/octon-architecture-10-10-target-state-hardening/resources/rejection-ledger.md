# Rejection Ledger

## Rejected option: architectural re-foundation

Rejected because the current class-root, authority, generated/read-model,
run/mission, support, and extension architecture is strong. The problem is not
wrong foundation; it is incomplete enforcement and proof.

## Rejected option: new top-level proof/control plane

Rejected because existing `state/evidence/**`, proof-plane framework roots, and
constitutional obligations are sufficient. A new control plane would violate
simplicity and boundary discipline.

## Rejected option: generated architecture maps as authority

Rejected because generated maps must remain read models. Authority belongs in
framework/instance and operational truth/evidence belongs in state roots.

## Rejected option: broad support widening during this packet

Rejected because the packet's purpose is to earn 10/10 quality for the current
architecture, not expand live support to non-live adapters or tiers.

## Rejected option: keep support artifacts flat

Rejected because target-state support claim hygiene requires physical and
mechanical partitioning by claim state.

## Rejected option: delete workflow compatibility immediately

Rejected because compatibility wrappers may still be necessary for migration.
They should be retained only with explicit retirement posture.

## Rejected option: solve maintainability by removing registries

Rejected because registry-backed structural modeling is load-bearing. The correct
move is generated maps and health gates, not a return to hand-maintained path
matrices.
