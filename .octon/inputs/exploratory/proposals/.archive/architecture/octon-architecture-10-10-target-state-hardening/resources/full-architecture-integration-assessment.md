# Full Architecture Integration Assessment

## Whole-system judgment

Octon's architecture is strongest when treated as a single constitutional runtime
system rather than independent subsystems. The target state must therefore close
integration gaps between:

- structural registry and active docs;
- root manifest and runtime config;
- support targets and support admissions;
- support admissions and capability-pack admissions;
- capability packs and execution requests/grants/receipts;
- extension activation and generated/effective publication;
- mission continuity and run contracts;
- run lifecycle and evidence store;
- proof-plane claims and disclosure artifacts;
- compatibility shims and retirement register.

## Integration invariants

1. No runtime route may depend on proposal paths.
2. No generated output may mint authority.
3. No input artifact may be a direct runtime or policy dependency.
4. No pack admission may widen support beyond a live tuple.
5. No mission may authorize material execution without a run contract.
6. No host/model adapter may widen authority.
7. No support claim may exceed proof bundle and disclosure coverage.
8. No generated/effective output may be runtime-consumed without freshness.
9. No compatibility shim may persist without owner and retirement trigger.
10. No closeout may succeed without evidence-store completeness.

## Integration risks

- Manual synchronization across registries can drift.
- Support state can be misunderstood if flat directories remain.
- Generated/effective outputs can become quasi-authority if freshness is weak.
- Packs can become a second support model if admissions are not tied to support targets.
- Boot complexity can hide which surfaces are actually required.

## Integration answer

The packet's proposed health contract is the integration point. Octon should not
create another control plane; it should create one health gate that validates the
existing control planes and refuses closure when they disagree.
