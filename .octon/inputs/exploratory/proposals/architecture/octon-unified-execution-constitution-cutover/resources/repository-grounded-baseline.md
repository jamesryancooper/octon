# Repository-Grounded Baseline

## What Octon actually is today
Octon is already a governed self-hosting harness built around a class-root super-root under
`/.octon/`.

### Strong current assets
- a clear authored-authority vs operational-truth vs derived-output split:
  - `framework/**`
  - `instance/**`
  - `state/**`
  - `generated/**`
  - `inputs/**`
- a fail-closed root manifest and runtime policy interface
- typed execution request / grant / receipt contracts
- a serious mission/autonomy policy surface with risk ceilings, oversight, safing, and breaker concepts
- continuity, evidence, and control roots already separated under `state/**`
- structural and governance CI that is stronger than most public harness repos
- newly introduced constitutional, lab, observability, support-target, and run-rooted surfaces

### Main current gaps and mis-boundaries
- workspace objective authority still resolves through bootstrap/cognition paths rather than a dedicated charter root
- host-native glue still matters too much in approval/review workflows
- run-rooted runtime is present but not yet universally proven on all consequential paths
- functional / behavioral / recovery proof is not yet as mature as structural / governance proof
- disclosure and replay exist but are not yet strong enough to support the highest target-state claim
- legacy cognition/persona surfaces remain in the interpretive path longer than the target design intends

## Baseline design consequences
1. Preserve the class-root super-root.
2. Consolidate constitutional authority.
3. Normalize execution around run contracts and run-rooted artifacts.
4. Centralize approval and revocation into first-class artifacts.
5. Keep structural proof strong while expanding the missing proof planes.
6. Simplify agency kernel surfaces.
7. Introduce real disclosure and replay gating.
