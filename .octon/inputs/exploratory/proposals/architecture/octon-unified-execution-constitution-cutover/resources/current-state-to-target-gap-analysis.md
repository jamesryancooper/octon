# Current-State to Target-State Gap Analysis

This document captures the packet's original design-time gap framing. For the
later repository-grounded implementation assessment added to this packet, read
`resources/unified-execution-constitution-audit.md` alongside this summary and
prefer the audit when current repository status matters.

## Gap 1 — repository constitution vs execution constitution
Octon already has a strong repository constitution. It does not yet universally guarantee that
all consequential execution is run-bound, authority-routed, proof-backed, replayable, and
fully disclosed.

## Gap 2 — mission-first bias
Mission is still too close to the canonical autonomy unit. The target state requires mission to
remain a continuity/ownership/autonomy container while run contracts become the atomic unit of
material execution.

## Gap 3 — host-shaped authority
GitHub labels and workflow glue still shape practical authority too directly. The target state
requires host adapters to become projections of canonical approval/grant/lease/revocation
artifacts.

## Gap 4 — proof-plane asymmetry
Structural and governance proof are stronger than functional, behavioral, maintainability, and
recovery proof.

## Gap 5 — evidence and disclosure normalization
RunCard and HarnessCard must become mandatory and sufficient for operational/scientific
interpretation. Large replay/telemetry payloads need disciplined externalization.

## Gap 6 — persona-heavy kernel residue
The target-state agency layer is simpler: an orchestrator-first kernel with only real
boundary-value roles and optional identity overlays.

## Gap 7 — portability realism
Support must be explicit and adapter-conformance-backed rather than implied.
