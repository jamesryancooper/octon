# Acceptance criteria

## Packet-level acceptance

- zero unresolved blockers for any concept retained as `adapt`
- two consecutive validation passes with no new blocking issues
- explicit retained evidence for every adapted capability
- no docs-only or proposal-only pseudo-capabilities classified as implemented
- no recommendation introduces a second control plane or shadow memory surface

## Per-concept acceptance criteria

### Progressive-disclosure context map

- confirm that the cited live repo anchors still materially embody the capability
- confirm no packet action would improve the repo more than the existing implementation

### Failure-driven harness hardening

- authoritative files created/edited in the candidate surfaces listed for this concept
- canonical control-state materialization exists: No standing new control plane; only proposal routing and optional run-local invocation state if operationalized later.
- retained evidence exists: Failure-distillation bundles linking source run evidence -> recurring pattern -> recommended authority updates.
- operator/runtime touchpoints are wired: Assurance jobs, recurring governance review, proposal generation, skills/context maintenance.
- validators/checks exist: New schemas, evidence bundle validation, regression proof that targeted failure classes recur less often after promotion.
- closure proof met: Two consecutive distillation cycles produce valid bundles; at least one promoted hardening closes a measured recurring failure class without shadow authority.

### Structured review findings + disposition

- authoritative files created/edited in the candidate surfaces listed for this concept
- canonical control-state materialization exists: Run-local review disposition state that can block stage/promote/finalize transitions.
- retained evidence exists: Retained findings, evaluator source, evidence refs, and disposition trace.
- operator/runtime touchpoints are wired: Evaluator agents, human review UI/process, run-controller gating, disclosure generation.
- validators/checks exist: Schemas, validator that unresolved blocking findings fail closed, proof that deferred concerns route into a durable follow-up surface rather than comments.
- closure proof met: Two consecutive validation passes show unresolved blocking findings prevent progression and deferred findings remain traceable.

### Reversible work-item state machine

- confirm that the cited live repo anchors still materially embody the capability
- confirm no packet action would improve the repo more than the existing implementation

### Thin adapters + token-efficient outputs

- authoritative files created/edited in the candidate surfaces listed for this concept
- canonical control-state materialization exists: No new mission control plane; optional repo-specific runtime profile only.
- retained evidence exists: Validation receipts proving output envelopes conform and full raw payloads remain recoverable outside the live context.
- operator/runtime touchpoints are wired: Adapter wrappers, tool runners, runtime prompt assembly, evidence publication.
- validators/checks exist: Schema validation plus token-budget checks in CI/runtime assurance.
- closure proof met: Two consecutive runs show compact envelopes under budget and no loss of recoverability for full payload evidence.

### Evidence bundles + observability

- confirm that the cited live repo anchors still materially embody the capability
- confirm no packet action would improve the repo more than the existing implementation

### Distillation pipeline from traces/comments/failures into proposal packets and shared skills

- authoritative files created/edited in the candidate surfaces listed for this concept
- canonical control-state materialization exists: No direct runtime dependency on distillation outputs until approved and promoted; any job scheduling state should stay in existing runtime/orchestration surfaces if later added.
- retained evidence exists: Input index, clustering rationale, proposed promotions, and post-promotion recurrence checks.
- operator/runtime touchpoints are wired: Recurring governance review, proposal authoring, context/skill maintenance, optional agent-assisted analysis jobs.
- validators/checks exist: Bundle schema, provenance checks, and proof that promoted distillations reduce recurrence without creating shadow memory.
- closure proof met: Two consecutive distillation passes produce valid bundles; at least one approved distillation is promoted into instance authority with traceable provenance and measurable value.

### Proposal-first mission classification

- authoritative files created/edited in the candidate surfaces listed for this concept
- canonical control-state materialization exists: Mission classification is live mutable control truth and must live under state/control/execution/missions.
- retained evidence exists: Proposal refs, acceptance basis, and validation showing policy was enforced before execution.
- operator/runtime touchpoints are wired: Mission intake, planner/orchestrator, approval workflow, run contract generation.
- validators/checks exist: Schema extension, validator that blocks execution without proposal when required, and policy tests across mission classes.
- closure proof met: Two consecutive validation passes show proposal-required classes fail closed when packet refs are absent and proceed correctly when packet refs are present.

### Selective dependency internalization

- explicit blocker/evidence gap recorded
- no architectural promotion attempted without a separate evidence-backed packet

### Unbounded domain access / approval bypass

- concept remains rejected
- current constitutional prohibition remains intact
