# Decision record plan

## Why decision records are needed

The actionable concepts in this packet do not require a new top-level architecture, but they do change the explicitness of several governance surfaces. That warrants decision capture when promoted.

## Recommended decision records if promotion proceeds

| Decision ID (proposed) | Scope | Decision |
|---|---|---|
| ADR-X1 | assurance | Extend evaluator review into finding/disposition pair rather than free-form review-only control |
| ADR-X2 | governance/evidence | Failure-distillation and evidence-distillation stay proposal-gated and may not auto-promote authority |
| ADR-X3 | agency runtime | Compact output envelopes are part of native-first adapter discipline |
| ADR-X4 | mission autonomy | High-ambiguity mission classes require proposal-first treatment and canonical mission classification control records |

## Decision posture

- If maintainers judge these changes to be local repository policy only, decisions may live under instance governance decisions rather than framework ADRs.
- If any shared framework contract becomes portable Octon core, the decision should be captured under framework governance decisions/ADR lineage.
- This packet does not pre-commit the exact ADR location; it only identifies the decisions that should be recorded.
