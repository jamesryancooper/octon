# Normalize Source Context

Read the supplied source material, preserve its complete relevant content under
packet `resources/**`, and identify whether it is an audit, architecture
evaluation, highest-leverage-next-step request, generic source input, drift
report, or program source.

Extract required source claims as traceable inputs: findings, gaps, target-state
thesis, score or closure target, repository-grounding requirements, requested
scope discipline, explicit non-goals, acceptance and closure conditions,
validation expectations, and any required evidence or downloadable/archive-ready
artifact expectation. Keep the full source available in resources even when the
packet narrows or rejects part of it.

Do not treat source material, manual prompt variants, or chat history as
authority. Re-ground all claims against the live repository.
